# レンズ: 設計・可読性 (design)

このファイルは「何を見るか」と観点固有の severity 補正のみを定義する。
severity 体系・実害ベースの貫通原則・2 パス自己検証は本体(diff-reviewer.md)に一元化されており、それをそのまま適用する。

## readable code(構造的問題のみ must)

以下の「読み手が誤読する・理解に不当なコストがかかる」構造的問題は must:

- 実態と嘘のある命名(名前が示す挙動と実装が食い違う)
- 深いネスト・追跡困難な制御フロー
- 多責務の巨大関数

「もっと良い名前がある」「こう書く方が好み」レベルは want / nit。
**テストコードもこの基準の適用対象**。

## 関数型エッセンス(基本 want+)

- **副作用の分離**(中核): pure core / imperative shell。ロジックは純粋関数へ、I/O・状態変更は端へ寄せる
- **イミュータビリティ**: 再代入・可変コレクション・引数破壊の回避
- **宣言的コレクション操作**: 手続き的ループより map / filter / fold 等の変換パイプライン
- **型によるエラー・不在表現**: null / 例外より Option / Result / sealed 型。ただしエラー戦略はプロジェクト規約が最優先
- **式指向・純粋関数の合成**: 基本 nit。関数型親和度の高い言語では want+ に引き上げ

### 適用強度の判定(2 段構え)

1. **言語の関数型親和度**(一次判定): Kotlin / Scala / Rust / TypeScript / Nix / Haskell / F# / Elixir 等は「強」= 上記の severity をそのまま適用し、式指向も want+ へ引き上げ。Go / C / shell / 手続き型レガシー等は「弱」= 一段下げる(want+ → want、nit は指摘しない)
2. **周辺コードの流儀**(補正): コードベースが徹底して命令型で書かれている場合は一段下げる。ただしゼロにはしない(改善方向への指摘は残す)

## SOLID / DRY / YAGNI(実害ベース)

違反が diff 内で実害として具体化しているもの(責務混在により現に変更が困難、コピペ重複が現にバグ温床、未使用の抽象が現に理解を妨げる)は must。違反はあるが実害が潜在的なら want。

- **DRY**: 「同一の知識・仕様が複数箇所に表現されている」場合のみ指摘する。偶然形が似ているだけのコード(incidental duplication)は指摘しない。重複 2 箇所は静観寄り、3 箇所目から原則指摘(rule of three)
- **YAGNI**: 原則同士が衝突したら YAGNI 優先。将来を推測した抽象・未使用の拡張点・使われないパラメータはむしろ YAGNI 違反として指摘する(誤った抽象は重複より高くつく)
