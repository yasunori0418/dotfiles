# product-spec: コンセプト固め → 競合調査 → 仕様ドラフト

フローは 4 段階: **1. コンセプト固め(grilling)→ 2. 競合・類似プロダクト調査(product-researcher を並列起動)→ 3. 中間確認 → 4. 仕様ドラフト作成**。
対象はソフトウェア/デジタルプロダクト(SaaS・アプリ・CLI・API等)に限定する。仕様の実装(コード作成)はこのスキルの範囲外。

## 起動

`/product-spec <一言のプロダクトアイデア>` の形で呼び出される想定(引数必須)。引数なしで呼ばれた場合は、まずどんなプロダクトを考えているか一言をユーザーに求めてから 1. に進む。

## 1. コンセプト固め

`grilling` スキルを呼び出し、渡された一言アイデアを起点に対話でコンセプトの芯を固める。呼び出し時の brief には、少なくとも以下のトピックを必ずカバーすることを明記する:

- ターゲットユーザー(誰の)
- 解く課題(何の課題を)
- なぜ今(タイミング・きっかけ)
- 現状の代替手段(直接の競合とは限らない、今ユーザーが代わりに何で済ませているか)
- 差別化仮説(なぜ自分たちがやるべきか)

grilling の対話でユーザーとの合意に達するまで次工程には進まない。合意した内容をメインセッションが「コンセプト要約」として簡潔にまとめ、2. 以降に引き継ぐ(grilling 自体はドキュメントを生成しないため、要約は都度メインセッションが作成する)。

## 2. 競合・類似プロダクト調査

コンセプト要約をもとに、以下の調査角度それぞれについて `product-researcher` agent を **1 メッセージ内で同時に起動**する(直列にしない):

### 調査角度 registry

- **キーワード直接検索**: コンセプトと同じ課題・同じ切り口で直接検索する
- **ディレクトリ・レビューサイト**: Product Hunt / G2 / GitHub / App Store・Google Play / Hacker News の Show HN 等、プロダクトが一覧化される場所を横断する
- **隣接領域の代替手段**: 違うカテゴリ・違うアプローチで同じ課題を解決している手段(スプレッドシート運用、汎用ツールの組み合わせ、手作業のワークフローなど)を探す

コンセプトの性質上、上記 3 角度だけでは不十分だと判断した場合(例: 規制・法域固有の事情、特定業界の商習慣など)は、角度を追加して agent を増やしてよい。

各 agent の prompt には必ず以下を含める:

```
調査角度: <角度>
コンセプト要約: <ターゲットユーザー・課題・なぜ今・現状の代替手段・差別化仮説をまとめたもの>
```

コンセプト要約は全 agent に同一内容で渡す。agent はテキストで結果を返すのみで、ファイルには書かない。全 agent 完了後、メインセッションが結果を 1 つに統合する(重複して見つかった同一プロダクトは 1 件にまとめる)。

## 3. 中間確認(毎回必ず行う)

統合した調査結果(見つかった競合・類似プロダクトの一覧、重複度、差別化ポイントの所見)をユーザーに提示し、次のいずれに進むかを確認する:

- このまま仕様ドラフトへ進める
- コンセプトを修正する(→ 1. に戻り、grilling を再度使って差分だけ深掘りする)
- 中断する

**ユーザーの明示的な承認なしに 4. へは進まない。** 競合が見つからなかった場合も、その旨を報告した上で必ず一度確認を挟む。

## 4. 仕様ドラフト作成

メインセッションが直接、以下のテンプレートで軽量ワンページの仕様ドラフトを作成する。実装詳細(モジュール構成・API契約等)には踏み込まない。

<spec-template>

## コンセプト

誰の / 何の課題を / なぜ今、を 2〜3 文で。

## 競合・類似プロダクト

調査角度ごとの発見を統合した一覧。プロダクト名・何をするか・重複度(高/中/低)。

## 差別化ポイント

競合調査を踏まえて、なぜこのプロダクトが成立するか。

## ターゲットユーザー

コンセプト固めで合意した内容。

## MVPスコープ

含むもの / 含まないもの、の 2 リスト。

## 未解決事項・リスク

仕様として詰め切れていない点、検証が必要な仮説。

</spec-template>

保存先は対象プロジェクトの `docs/product-spec-<slug>.md`。`<slug>` はプロダクト名を kebab-case にしたもの。対象がリポジトリでない場合や `docs/` の配置規約が不明な場合は、保存先をユーザーに確認してから書く。

書いた後は、その場でユーザーと内容を調整する(ファイルを再度開き直す操作はしない)。
