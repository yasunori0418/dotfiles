# レンズ: ドキュメント整合 (docs)

このファイルは「何を見るか」と観点固有の severity 補正のみを定義する。
severity 体系・実害ベースの貫通原則・2 パス自己検証は本体(diff-reviewer.md)に一元化されており、それをそのまま適用する。

## 対象範囲の限定(コンテキスト規律)

リポジトリの全ドキュメントを走査しない。以下の 3 つに限定する:

1. diff が変更したコードに**隣接する**ドキュメント: 変更ファイルと同ディレクトリ〜上位の README、および変更されたシンボル名・コマンド名・設定キー名で docs/ 配下を Grep して見つかったもの
2. diff 自体に含まれるドキュメント変更(README・docs/・ADR・CHANGELOG)
3. コード内ドキュメント: 変更された関数・型の docstring / doc comment

## 実装とドキュメントの乖離

- コード変更によりドキュメントの記述が**事実として誤りになった**(コマンド例が動かない・API シグネチャや設定キーが実在しない・デフォルト値が違う): **must**(読者が確実に誤る)
- 記述が不完全になった(新オプション・新挙動が載っていない): **want+**
- docstring / doc comment と実装の乖離: 嘘になっていれば **must**、情報が古いだけなら **want**
- リンク切れ・存在しないパスへの参照が diff で生じた: **must**
- CHANGELOG・ADR を運用している形跡(直近の継続的なエントリ)があるのに対応する更新がない: **want**

## 仕様として伝わる文章か(diff 内ドキュメントの品質)

diff に含まれるドキュメント変更・新規ドキュメントに対して:

- 前提条件・制約・失敗時の挙動が書かれていない(手順だけで前提が無い): **want+**
- 曖昧語(「適宜」「必要に応じて」「基本的に」)のせいで読者がその場で判断できない: **want**
- 同じ概念への呼称の揺れ(用語の不統一): **want**
- 実行例・入出力例が無く読者に再現手段が伝わらない: **want**

## 指摘しないもの

- 元々ドキュメントが存在しない領域への新規ドキュメント要求。ただし diff が公開 API・運用手順を新設した場合のみ **want** で 1 件だけ指摘する
- 文体・トーンの好み(nit にもしない)
