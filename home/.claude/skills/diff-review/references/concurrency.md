# レンズ: 並行性 (concurrency)

このファイルは「何を見るか」と観点固有の severity 補正のみを定義する。
severity 体系・2 パス自己検証は本体(diff-reviewer.md)に一元化されており、それをそのまま適用する。

## severity 判定(貫通原則の読み替え)

本体の貫通原則(実害が diff 内で具体化しているか)を「**並行実行経路が現に存在し、競合の実害が成立しているか**」と読み替えて適用する:

- **must**: 変更されたコードに複数スレッド / リクエスト / コルーチンが現に到達し(到達性を呼び出し元で確認する)、データ破壊・デッドロック・二重実行が成立
- **want+ / want**: 並行到達が構成次第・将来的、現状は単一実行だが保護も無い
- この観点は誤検知が多い。**反証パスを特に厳格に**: シングルスレッド実行が保証される文脈(イベントループ・アクター・リクエストスコープの新規インスタンス)でないかを必ず確認し、確証が持てない指摘は [PLAUSIBLE] を付けて残す

## 共有状態の競合

- 共有可変状態(フィールド・グローバル・シングルトン・キャッシュ)への非同期アクセス
- check-then-act(存在確認してから作成、読んでから更新)の非アトミックな複合操作(TOCTOU)
- 非スレッドセーフなコレクション・オブジェクトの並行共有

## デッドロック・ロック運用

- 複数ロックの取得順序の不整合
- ロック保持中の外部呼び出し(I/O・コールバック・ユーザーコード)による保持時間の伸長
- ロック粒度の問題は棲み分け: 正しさ(粗すぎて守れていない)は concurrency、速度(直列化ボトルネック)は performance

## async / await の誤用

- async コンテキスト内のブロッキング呼び出し(イベントループ・ディスパッチャの停止)
- fire-and-forget の例外握り潰し(await されない Promise・join されないコルーチン/goroutine)
- await / join 漏れにより完了前に後続処理へ進む経路

## 分散・DB の並行性

- 同時実行されうる処理の冪等性欠如(リトライ・多重リクエストで二重課金・二重送信)
- lost update: 読み取り値に基づく更新に楽観ロック(バージョン列)も悲観ロックも無い
- トランザクション分離レベルと実装の期待の不整合、トランザクション境界の外にある一貫性依存
