# レンズ: 可観測性・運用 (ops)

このファイルは「何を見るか」と観点固有の severity 補正のみを定義する。
severity 体系・2 パス自己検証は本体(diff-reviewer.md)に一元化されており、それをそのまま適用する。

## severity 判定(貫通原則の読み替え)

本体の貫通原則(実害が diff 内で具体化しているか)を「**障害時に調査不能・運用不能になる経路が diff 内で成立しているか**」と読み替えて適用する:

- **must**: 失敗が痕跡を残さず握り潰される経路(空 catch・エラー無視で処理継続)、外部呼び出しの無限待ちが diff 内で成立
- **want+ / want**: ログはあるが文脈不足、レベル不適切、計測の欠落
- 秘匿情報・PII のログ出力は security レンズ(A09)の領分。ops は「情報が足りない・役に立たない」側を見る

## ログ

- エラーハンドリング箇所に原因究明へ必要な文脈があるか: 対象の識別子(ID・キー)・入力の要約・元例外(cause の連鎖を切らない)
- レベルの適切性: 定常系を warn / error で出す(ノイズで本物の警報が埋まる)、異常系を info / debug で出す(検知漏れ)
- 高頻度ループ・リトライ内の無条件ログ(ログ流量の爆発)

## エラーメッセージ

- 「何が・どの対象で・どうすれば直るか」が入っているか。「failed」「error occurred」だけの文言は **want+**
- ユーザー向けと運用者向けの混同: 内部詳細の露出は security 側、運用者が必要な情報の欠如が ops 側

## メトリクス・トレース

- 計測基盤(Prometheus・OpenTelemetry・Datadog 等)の有無をプロジェクトの実態で判定する
- 基盤あり + 新規の外部呼び出し・重要な処理経路の追加: レイテンシ・成功率の計測欠落は **want**
- 基盤なし: 要求しない

## 運用性

- 外部呼び出し(HTTP・DB・キュー)のタイムアウト未設定: 無限待ちの経路が diff 内で成立していれば **must**
- リトライの上限・バックオフ欠如(無限リトライ・thundering herd)
- graceful shutdown を壊す変更(シグナル処理・in-flight 処理の破棄・リソースの後始末漏れ)
- 影響の大きい新機能に対する feature flag / kill switch の欠如: flag 基盤が既に運用されている場合のみ **want**
