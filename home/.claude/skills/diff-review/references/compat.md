# レンズ: 互換性・移行安全 (compat)

このファイルは「何を見るか」と観点固有の severity 補正のみを定義する。
severity 体系・2 パス自己検証は本体(diff-reviewer.md)に一元化されており、それをそのまま適用する。

## severity 判定(貫通原則の読み替え)

本体の貫通原則(実害が diff 内で具体化しているか)を「**公開済みの契約の破壊が diff 内で成立しているか**」と読み替えて適用する:

- **must**: 外部に公開済みの契約(API・スキーマ・CLI・設定・イベント)の破壊的変更が、バージョニング・移行手順・後方互換措置なしに diff 内で成立
- **want+ / want**: 利用者が社内・少数に限られる、互換措置はあるが移行手順が暗黙、破壊がまだ潜在的(未リリース経路)
- 未公開・開発中の契約の変更は指摘しない(公開前の作り直しはむしろ健全)

## API 互換

- シグネチャ・レスポンス形状・ステータスコード・エラー形式の破壊的変更
- 必須パラメータの追加・enum 値の削除・フィールドの意味変更(既存クライアントが壊れる)
- バージョニング運用(パス・ヘッダ)が存在するのに破壊変更が同一バージョンに入っている

## DB マイグレーション

- 旧コードと新スキーマが共存できない変更(カラム削除・リネーム・型変更・NOT NULL 追加)を 1 段で行う: ローリングデプロイ・ロールバックが壊れる。expand-contract(追加 → 二重書き込み/移行 → 削除)への分割を修正方針として示す
- ロールバック不能なマイグレーション(データ破壊を伴う・down が無い/嘘)
- 大テーブルへのロックを伴う DDL・全件 UPDATE(オンライン実行可否の確認)

## 直列化・メッセージ互換

- イベント / メッセージスキーマの破壊変更(コンシューマの対応が diff に無い)
- キャッシュ・セッション・キューに滞留中の旧形式データが新コードでデシリアライズできない
- 未知の enum 値・フィールドに対するハンドリングの欠如(前方互換の欠落)

## 設定・運用互換

- 設定キー・CLI フラグ・環境変数のリネーム / 削除(エイリアス・非推奨期間なし)
- デフォルト値の変更で既存環境の挙動が黙って変わる: 変更自体ではなく、移行手順・告知(CHANGELOG 等)の欠如を指摘する
- ライブラリの場合: semver 運用があるなら破壊変更と版数の不整合
