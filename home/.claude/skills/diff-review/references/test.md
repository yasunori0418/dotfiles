# レンズ: テスト設計・網羅性 (test)

このファイルは「何を見るか」と観点固有の severity 補正のみを定義する。
severity 体系・実害ベースの貫通原則・2 パス自己検証は本体(diff-reviewer.md)に一元化されており、それをそのまま適用する。

## 品質モデル(検証対象の切り分け)

ISO/IEC 25010 の区分で、このレンズが検証するのは**外部品質**(実行時に観測される振る舞いが仕様に適合しているか)。
**内部品質**(コード構造・保守性)は design レンズの領分で、ここでは扱わない(テストコード自体の可読性・構造も design が見る)。
テストは外部品質を固定する装置: 入出力・観測可能な振る舞いを検証し、内部構造には結合しない。これが後述「実装詳細への結合」を退ける根拠。

## テスト基盤の有無の判定

言語で一律に判断せず、プロジェクトの実態で判定する: 既存テストファイル(`*_test.go`・`*.test.ts`・`spec/`・`tests/` 等)、`flake.nix` の `checks` 出力、CI のテスト workflow、テストランナー設定の存在。
Nix プロジェクトでも flake checks 等でテストを持つものがある。

## テスト無し変更

- 基盤あり: diff にロジック変更があるのに対応するテストが無ければ **must**。指摘には追加すべきテストケース(正常系・境界値・異常系)を具体的に列挙する
- 基盤なし: 「テスト基盤ごと導入すべき」を **want** で 1 件だけ指摘。テストを書く現実的手段が無い対象(純粋な設定ファイル等)では発火させない
- 挙動に影響しない変更(rename・ログ・コメント・設定値)にはテストを要求しない

## ケース網羅の欠落(テスト設計技法で導出)

変更したロジックの分岐構造に対し、ISTQB のテスト設計技法で欠けを体系的に特定する:

- **同値分割**: 入力の同値クラス(有効・無効)ごとに最低 1 ケースあるか
- **境界値分析**: 分岐の境界(0 / 1 / 最大 / 空 / null / 境界の両側)が検証されているか
- **デシジョンテーブル**: 複数条件の組み合わせで挙動が変わる箇所で、有意な組み合わせが揃っているか
- **状態遷移**: 状態を持つ変更で、正当な遷移と不正な遷移(ガード)の両方が検証されているか

主要分岐(エラー処理・境界値)の未検証は **must**、枝葉のケースは **want**。指摘時はどの技法でどのケースが欠けているかを明示する。

## テストレベルの適切性(テストピラミッド)

- 単体テストで検証できる純粋なロジックが統合 / E2E テストでしか検証されていない: **want+**(遅い・不安定・原因特定が困難)
- 結合部(I/O・設定・配線)の検証をモックだけで済ませ、結合レベルの検証が存在しない: 実害(現に結合バグを検知できない構図)が diff 内で具体化していれば **want+**、潜在的なら **want**

## テストプロセス観点(ISTQB 7 原則の diff レビュー適用)

- **欠陥の偏在**: 変更が集中しているファイル・修正が繰り返されてきた箇所(`git log --follow` で確認可)は欠陥密度が高いとみなし、網羅要求を一段厳しくする
- **殺虫剤のパラドックス**: 挙動を変えたのに既存テストが 1 つも変更・追加されていない場合、既存テストがその挙動を検知できていないシグナルとして疑う。リグレッション検知力の欠如が確認できたら **want+**

## 実装詳細への結合

モック過多・内部状態や呼び出し順の検証など「リファクタすると壊れるテスト」は基本 **want+**。リファクタ阻害が現に具体化していれば must。
振る舞い(入出力)ベースへの誘導として「純粋関数化すればモックが不要になる」という副作用分離との接続を修正方針に使う(根拠は冒頭の品質モデル)。
