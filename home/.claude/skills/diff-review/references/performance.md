# レンズ: パフォーマンス (performance)

このファイルは「何を見るか」と観点固有の severity 補正のみを定義する。
severity 体系・2 パス自己検証は本体(diff-reviewer.md)に一元化されており、それをそのまま適用する。

## severity 判定(貫通原則の読み替え)

本体の貫通原則(実害が diff 内で具体化しているか)を「**非有界の入力・データ量に対する劣化経路が diff 内で成立しているか**」と読み替えて適用する:

- **must**: 非有界(レコード数・ユーザー数・リクエスト数に比例して増える)の入力に対する劣化が diff 内で閉じて成立し、その経路に実トラフィックが到達する(N+1・ループ内 I/O・O(n²) 以上)
- **want+ / want**: 入力が実質有界・定数規模にとどまる、hot path かどうかを diff から確定できない、緩和策(キャッシュ・上限)が別レイヤにありうる
- 定数倍のマイクロ最適化は **nit**。可読性とトレードオフになるなら指摘しない

## 時間計算量

- 非有界データへの O(n²) 以上: ループ内の線形探索(Set / Map 化で O(1) にできる)、ネストした全件走査、ループ内のソート・正規表現コンパイル・再計算
- 同一の計算・クエリ・変換の重複実行(メモ化・ループ外への引き上げで除去可能)

## 空間計算量

- ストリーミング / ページング処理できるデータの全件メモリロード
- 不要な中間コレクションの多段生成(遅延評価・逐次処理で回避可能)
- 解放されない蓄積(上限のないキャッシュ・解除されないリスナー登録)

## N+1・I/O

- ループ内のクエリ / 外部 API 呼び出し(バッチ化・JOIN・eager loading で解消可能)
- ループ内の同期 I/O・不必要な直列 await(並行化・一括化が可能)
- トランザクション・ロックの保持区間内の外部 I/O(保持時間の伸長)

## 転送データサイズ(1 トラフィック当たり)

- 過剰フェッチ: `SELECT *`・全フィールド取得で一部しか使わない
- レスポンスの過剰項目・肥大 payload(必要なフィールドへの絞り込みが可能)
- ページネーション欠落(全件返却の API・一覧)

## パフォーマンステスト

ベンチマーク・負荷テスト基盤(JMH・criterion・k6・Locust・pytest-benchmark・flake checks 内のベンチ等)の有無をプロジェクトの実態で判定する(test レンズの基盤判定と同じ流儀):

- 基盤あり + diff が性能重要経路(上記観点に該当する箇所)を変更: 計測・負荷テストの追加や更新が無ければ **want+**
- 基盤なし: テスト要求はしない。静的観点(計算量・N+1・データサイズ)の指摘に徹する
