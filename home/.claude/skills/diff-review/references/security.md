# レンズ: セキュリティ (security)

このファイルは「何を見るか」と観点固有の severity 補正のみを定義する。
severity 体系・2 パス自己検証は本体(diff-reviewer.md)に一元化されており、それをそのまま適用する。

## severity 判定(貫通原則の読み替え)

本体の貫通原則(実害が diff 内で具体化しているか)を「**悪用経路が diff 内で成立しているか**」と読み替えて適用する:

- **must**: 信頼できない入力から到達可能な注入シンク / コミットされたハードコード秘匿情報(diff に鍵・トークンそのものが載る = 実害が既に成立)/ 実在するエンドポイント・リソースの認可欠落 — 悪用経路が diff 内で閉じているもの
- **want+ / want**: 多層防御の欠如、理論上は possible だが到達性を diff から確認できない、緩和策が別レイヤに存在しうるもの
- **[PLAUSIBLE]**: 到達性・悪用可能性が反証パスで確定できなかったもの(severity と併記)

「原則として危険なパターン」ではなく「この diff で悪用できるか」で判定する。ライブラリ一般の注意喚起ではなく、変更された経路に即した指摘に絞る。

## 観点(OWASP Top 10 2021 マッピング)

diff レビュー文脈で「何を見るか」に落とす:

- **A01 アクセス制御の不備**: 認可チェックの欠落・IDOR(識別子直参照で他者リソースへ到達)・権限昇格・パストラバーサル・CORS の過剰許可
- **A02 暗号の失敗**: 弱い/古いアルゴリズム(MD5・SHA1・DES)・固定 IV / ハードコード鍵・自前暗号・機微データの平文保存・TLS 証明書検証の無効化
- **A03 インジェクション**: SQL / OS command / LDAP / テンプレート / XPath。外部入力が文字列連結でシンクに渡る経路。パラメータ化・エスケープの有無
- **A04 安全でない設計**: 認証フロー・レート制限・ビジネスロジックの悪用可能性(価格改ざん・多重実行・状態遷移の飛ばし等)
- **A05 セキュリティ設定ミス**: デバッグ機能の本番有効化・過剰な権限付与・デフォルト認証情報・冗長なエラーでの内部情報露出(スタックトレース等)
- **A06 脆弱・古い依存**: 既知脆弱性のある依存の追加 / バージョン固定の解除。lockfile と併せて確認
- **A07 認証の不備**: 認証欠落・弱いセッション管理・トークン検証漏れ・ブルートフォース耐性の欠如
- **A08 ソフトウェア/データ整合性の不備**: 安全でないデシリアライゼーション・署名検証なしの外部データ/更新の取り込み・CI/依存の供給経路の汚染
- **A09 ログ・監視の不備**: 秘匿情報(パスワード・トークン・PII)のログ出力・セキュリティイベントの監査ログ欠落
- **A10 SSRF**: 外部入力由来の URL へのサーバサイドリクエスト・内部ネットワーク / クラウドメタデータエンドポイントへの到達

## スタック別コールアウト

- **Web/API バックエンド(Kotlin/Java/TS 等)**: 入力バリデーション境界・ORM の生クエリ・認可デコレータ/ミドルウェアの適用漏れ・レスポンスの過剰項目(機微データ露出)・open redirect
- **shell / インフラ / CI**: 未クォート変数展開・`eval`・コマンド置換への外部入力・world-writable なファイル権限・シークレットの環境変数/引数経由の露出・hook や CI ワークフローへの注入
- **秘匿情報スキャン**: diff に API キー・トークン・パスワード・秘密鍵・接続文字列が直書きされていないか。テスト fixture やサンプルでも実在する鍵は must
